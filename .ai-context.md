### Business_Rules
- BR-01 Net balances only: settlements must be computed from net member positions.
- BR-02 Monetary integrity: all money values must be integer cents; no floating point arithmetic.
- BR-03 Positive amounts: expenses and installments must be greater than zero.
- BR-04 Membership scope: a user can only read or mutate data from households where they are an active member.
- BR-05 Privileged actions: settlement closing and split-config changes require owner/admin role.
- BR-06 Referential integrity: each expense must reference valid household, payer member, and category.
- BR-07 Soft deletion policy: member/card/expense deletion must preserve financial traceability.
- BR-08 Deterministic settlement period: settlement must be reproducible for a specific period window.
- BR-09 Auditability: sensitive financial changes must be traceable (who, when, before, after).
- BR-10 API contract stability: error codes must remain stable and machine-readable.

## BUSINESS_INVARIANTS
- Never bypass domain constructors for entities with invariants.
- Never allow cross-household access without explicit authorization checks.
- Never compute settlements with float math.
- Never accept `amount_cents <= 0` in create/update flows.
- Never execute privileged household actions without owner/admin authorization.
- Never return untyped or unstable error payloads from API handlers.
- Never remove financial records in a way that breaks historical traceability.
- Never ship schema changes without matching migration, test, and rollback note.

## DATA_SCHEMA_REFERENCE
- `users` (auth identity) can be linked to many `members` records across households.
- `households` is the aggregate root for collaboration scope.
- `members` belongs to one household and can have role/salary/split responsibility.
- `expenses` belongs to one household and references payer member; may reference category/card.
- `installments` belongs to one expense for MSI or split temporal debt representation.
- `recurring_expenses` stores templates that generate concrete expenses by schedule.
- `categories` belongs to one household and classifies expenses.
- `cards` belongs to one household and models payment instrument metadata.
- `household_split_config` assigns proportional split percentages per member.
- Settlement model:
  - Input: members + expenses + installments for period.
  - Output: member balances and transfer recommendations.
  - Constraint: outputs must satisfy conservation of value (sum balances = 0).

## DEFINITION_OF_DONE
- Architecture
  - Domain layer has no HTTP, DB, or infrastructure imports.
  - Use cases depend only on domain and ports.
  - Adapters implement ports and keep business logic out of handlers.
- Code Quality
  - New code compiles and passes package tests.
  - Errors are wrapped with context and mapped to stable API error codes.
  - No obvious security regressions (authz, validation, secrets handling).
- Testing
  - Backend: unit tests for use case changes and handler contract tests for API changes.
  - Backend critical data paths: integration tests for postgres repositories.
  - Frontend: component/integration tests for changed flows.
- API Contract
  - JSON success and error shape remain consistent.
  - Status codes are semantically correct (400/401/403/404/409/500).
- Data
  - Any schema change includes migration + rollback strategy note.
  - Constraints/FKs/indexes align with new queries and invariants.
- Operational
  - Security defaults reviewed for production (`JWT_SECRET`, CORS, TLS/sslmode, rate limit).
  - Logs include request context without sensitive data leakage.
